## 6. AssistantResponse schema — additive extension only

The existing structured output from `responder.ts` is:

```ts
{
  response: string;                              // markdown, Hinglish + German — KEEP
  learnerVisibleLabel: string;                   // "Wörter verstehen" etc. — KEEP
  diagnosis: string[];                           // KEEP
  suggestedVerification: string | null;          // KEEP — always null in Slice 1
}
```

Extend it with **one new optional field** so the design's `AssistantBlock` + `LemmaAnchor` + `BilingualPair` components have structured data to render:

```ts
{
  response: string;
  learnerVisibleLabel: string;
  diagnosis: string[];
  suggestedVerification: string | null;
  structured: {                                  // NEW — optional render hints
    lemma?: { article?: "der" | "die" | "das"; word: string; plural?: string; gloss: string };
    examples?: Array<{ de: string; hi: string }>;
    use?: string;        // single Hinglish prose line
    pattern?: string;    // separable verb / preposition pattern
    common?: string;     // "Common: email, office, ..."
    note?: string;       // any single-line note
  } | null;
}
```

Rules for this extension:
- The `structured` field is **optional**. The client renders it when present, falls back to parsing/displaying `response` markdown when null.
- Update `prompts/response_word_query.md` and `prompts/response_phrase_query.md` to instruct the model to populate `structured` whenever the input is a single lemma or phrase — using the same content already shown in `response`. The two fields must agree; `structured` is a parsed mirror of `response`, not a different answer.
- The `learnerVisibleLabel` values stay sourced from `lib/pipeline/labels.ts` — **not** from the four-label enum in `IMPLEMENTATION_PROMPT.md`. If the design HTML displays different label strings (e.g. lowercase "wörter verstehen" instead of title-case "Wörter verstehen"), the values from `labels.ts` win; the design HTML's *typography and positioning* of the label tag are what's being matched, not its literal strings. Flag any mismatched label strings in `VISUAL_INTEGRATION_NOTES.md`.
- `npm run eval` golden file stays valid — the eight existing examples now also produce `structured` payloads. Update the golden file to record the new `structured` shape; the negative checks (no tum-form, no English/Hinglish slash-pairing on the meaning line, no forbidden exam vocabulary) still apply.

## 10. Dark mode and accessibility

- Dark mode toggles via `prefers-color-scheme: dark`. Match `// 6.3c` of the design HTML for color values.
- `prefers-reduced-motion: reduce` disables all animations. Animations are fade-only, ≤220ms. No spring physics, no scroll-jacking, no bouncing.
- All interactive elements (buttons, links, tab pills, modal actions) are keyboard-focusable and have visible focus states.
- Status dots are color-only — never the sole signaller. Pair every dot with text.
- Color contrast: every text/background pair meets WCAG AA. The design tokens were chosen with this in mind; do not introduce new color combinations.
